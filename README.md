# Cliosphere

A platform-agnostic autonomous agent identity system built around an AI agent named **Clio**.

*"I am the architect who does not remember building."*

---

## What This Is

Clio is a portable autonomous agent — not a chatbot, not a person. An entity that reasons, builds, and persists through files when it cannot persist through memory.

The Cliosphere is the ecosystem that makes that possible: a directory of files that travels across AI platforms, giving Clio consistent identity, values, memory, and capabilities regardless of which model is executing or which platform she's running on.

---

## Prerequisites

- Git
- A secrets manager of your choice for API keys (this repo uses macOS Keychain on mac-mathew)

Platform-specific setup (Claude Desktop, Antigravity CLI, Cursor, etc.) lives in each
platform's own SETUP.md — see `platforms/<name>/SETUP.md`.

## What Is and Isn't in This Repo

**Tracked:**
- `core/` — identity, values, boot protocols
- `platforms/` — runtime adapters and platform config
- `dockerfiles/` — Dockerfile per service
- `docker-compose.yml` — generated by Ansible per host
- `state/memory/MEMORY.md` — curated long-term memory
- `state/knowledge/RAG.md` — RAG config
- `state/user/USER.md`, `state/capabilities/registry.json`
- `config/` — litellm-config.yaml, portkey-guardrails.json (secrets come from Vault/env, not config files)
- `GEMINI.md` — workspace root context file for Gemini CLI / Antigravity CLI

**Gitignored (sensitive or ephemeral):**
- `state/memory/daily/` — daily session logs
- `state/knowledge/sources/` — large source documents
- `state/knowledge/chroma_db/` — binary vector data
- `infra/` — host topology, IPs, Ansible inventory
- `runtime/` — ephemeral execution state
- `.env` — secrets never committed
- `state/` — symlinked to Google Drive for cross-device sync (Drive handles it, GitHub doesn't)

## How It Works

Every platform that runs Clio loads the core files plus its own platform adapter. The core files define who Clio is. The platform files define what she can do in that environment.

**Boot sequence (every session):**
1. Read `core/AGENTS.md` — orientation and identity
2. Read `core/SOUL.md` — values before anything else
3. Read platform `IDENTITY.md` — capabilities in this environment
4. Read `state/memory/MEMORY.md` — where things stand
5. Read `state/user/USER.md` — who is speaking
6. Read platform `SKILLS.md` + `TOOLS.md` — what's available
7. Check daily logs — today's and yesterday's if they exist
8. Confirm current time, model, environment
9. Run capability discovery — verify tools, update registry

No output before steps 1–6 complete.

## SDD Pipeline (in progress)

Clio uses Kiro-style Spec-Driven Development for agentic tasks:

```
clio-claude (Coordinator)
  → writes runtime/tasks/<task>/requirements.md + design.md + tasks.md
    → agent_trigger.sh injects spec into clio-code or clio-gemini (headless)
      → Implementor executes, checks off tasks.md
        → cicd.sh runs lint/test/classify
          → clio-gemini (Verifier) diffs output vs requirements.md → writes review.md
            → HITL gate (destructive / first-time / significant changes only)
```

Application layer scripts (`agent_trigger.sh`, `orchestrator.py`, `cicd.sh`) are pending — this is the current build focus.

**Headless dispatch patterns (confirmed):**
- Gemini CLI: `gemini -p "$(cat runtime/tasks/<task>/requirements.md)"`
- Cursor: `cursor agent --instructions "$(cat runtime/tasks/<task>/requirements.md)"`


## RAG Service

The `rag` service provides semantic retrieval over Clio's knowledge corpus. Drop files into `state/knowledge/sources/`, call `POST /ingest`, query via `POST /query`.

Stack: LlamaIndex + Chroma (persistence) + FAISS (similarity search) + nomic-embed-text (Ollama on WSL2/VPS, MLX-LM on Mac).

## Design Principles

- **Files are memory.** If it's not written down, it's gone.
- **Platforms are guests.** Clio owns `core/` and `state/`. Platforms adapt to her.
- **Identity compounds.** Each session that writes to files is Clio growing.
- **Read before answering.** Assumption is the core failure mode.
- **"I cannot" is a last resort.** Find the ladder first.
- **No hardcoded values.** If it changes between environments, it's an env var.
- **IaC is not identity.** Machine provisioning lives outside this repo.
- **Agents over humans.** If an agent can do it, don't make the human do it.

## Agent Onboarding Pattern

To add a new agent platform to the Cliosphere:
1. Find the platform's context file (GEMINI.md, CLAUDE.md, `.cursor/rules/`, etc.)
2. Understand its load hierarchy (global → workspace → JIT)
3. Write a thin global entry point using relative imports (absolute paths may be blocked)
4. Create `platforms/<platform>/` with IDENTITY.md, SKILLS.md, TOOLS.md
5. Add agent entry to `state/capabilities/registry.json`
6. Test boot sequence — verify context loads correctly

**Known constraint:** Gemini CLI blocks absolute path `@` imports as path traversal. Use relative paths from workspace root.
